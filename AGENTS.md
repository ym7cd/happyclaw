# HappyClaw — Codex 协作说明

本仓库基于 GitHub 上游项目 fork 而来。当前上游维护方向不计划支持 Codex，因此本仓库在保持原有产品设计和工程风格的前提下，承担 Codex 相关能力的二次开发与长期维护。

这意味着后续涉及 Codex 的改动默认遵守以下额外约束：

- **拒绝一次性改动**：不接受只为当前问题打补丁、后续无法复用或难以继续演进的实现。
- **尽量与原设计对齐**：优先沿用上游已有抽象、目录结构、数据流和交互方式，不轻易引入平行体系。
- **保持可持续跟随上游**：方案应尽量缩小 fork 差异面，便于后续吸收上游优化与修复。
- **分支演进也要像上游特性**：即使是 fork 私有能力，也应达到“如果未来上游愿意接收，这个设计本身站得住脚”的标准。
